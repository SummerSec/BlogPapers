# AI Agent 工程的必然演进：CLI、Skills、Harness 与 Loop

这里持续记录我对 AI 的观察、判断和想法。

## 想法记录

### AI 编程工具的演进不是偶然

过去一年，我深度使用了 Claude Code、Codex 等各种 Vibe Coding 工具。我逐渐发现，Skills、Hooks、MCP 服务、各种 Loop，以及最近受到关注的驾驭工程（Harness Engineering），并不是凭空出现的零散功能或短期热点。它们背后对应的是 AI 编程工具发展的不同阶段。

从历史发展的角度看，这些能力都是必然的产物：当模型能力、上下文规模、工具调用方式和人的实际需求发展到某个阶段，相应的产品形态自然就会出现。理解它们不能只看单个功能，而应该把它们放回 AI 编程工具的演进过程里，看清每个阶段试图解决的问题。

### MCP 只是一个必然被淘汰的中间产物

不过，并非每一种阶段性产物都会成为最终形态。以 MCP 服务为例，我认为它并不算特别优雅。在许多真实的工程场景中，它本质上没有成熟的 CLI 命令行工具好用。CLI 更直接，能力边界更清楚，也更容易被人和 Agent 共同调用、组合、调试和复用。

MCP 的阶段性价值，在于 Agent 还不能充分驾驭 CLI 和现有工程工具时，统一工具接入方式、降低连接成本。但它只是 CLI 被 Agent 普遍、稳定地使用之前的一种过渡产物。随着 Agent 对终端、操作系统和现有工程工具的驾驭能力继续增强，原本需要专门封装成 MCP 服务的能力会重新回到更通用、更原生的 CLI 工具链中。因此，我认为 MCP 这种服务形态最终必然会被淘汰。

更粗暴一点说：在大家还没有意识到 CLI 才是 AI 使用工具的最佳形态之前，MCP 必然会作为中间产物存在；一旦这种认识成为共识，MCP 就不可能继续作为主流形态长期存在。这是一个相当直接、甚至有些武断的观点，但它可以作为观察后续技术演进的基本假设。

### 为什么 MCP 会被淘汰

#### 1. 朴素的 MCP 实现会长期占用模型上下文

MCP 的第一个现实问题，是许多客户端会把工具定义和能力说明直接暴露在模型上下文中。无论当前任务是否需要调用这些服务，模型都要带着相关信息工作。这会占用大量宝贵的上下文资源；当接入的 MCP 服务越来越多时，也会增加模型理解、筛选和决定使用哪个工具的负担。

Skills 后来体现出的管理理念值得借鉴：能力不必始终展开在上下文里，而应当按需加载。CLI 可以很好地配合这种方式。我们可以把 API 的说明、参数和调用方法写进 Skill 文件，由 Skill 在需要时读取说明、包装流程，再调用相应的 CLI 工具。模型平时只需要知道某项 Skill 存在，不需要始终携带它背后的全部接口定义。

相比之下，`Skill + CLI` 把工具知识从常驻上下文变成了按需获取的外部知识，既减少上下文消耗，也能让能力的组织、维护和组合更加灵活。这种渐进式披露的方式，比让大量 MCP 工具长期暴露在上下文中更符合 Agent 的发展方向。

不过，这里必须区分 MCP 协议和客户端实现。MCP 本身提供了 `tools/list`、分页以及工具列表变更通知，客户端完全可以先发现工具，再按需把相关定义交给模型。MCP 官方的客户端最佳实践也把渐进式发现作为解决大量工具浪费 token、增加延迟和降低模型表现的方法。因此，“所有 MCP 工具始终暴露在上下文中”并不是协议强制要求，而是部分客户端采用的朴素实现。

这条反证意味着：上下文膨胀可以用来批评当前 MCP 的常见使用方式，却不足以单独证明 MCP 必然被淘汰。`Skill + CLI` 的渐进式披露仍然更直观，但 MCP 客户端也可以吸收同样的理念。要论证 MCP 最终会消失，还需要成本、可组合性、调试体验、权限边界和生态复用等方面的更多理由。

#### 2. CLI 可以复用成熟的工程体系

CLI 的另一个优势，是它可以在工具内部完整处理鉴权、状态和运行环境。它的输入输出也可以根据具体场景自由设计，不必先迁就一套额外的 Agent 协议。相比之下，MCP 服务需要遵守协议规定的工具描述、输入输出结构和传输方式。这些约束带来了兼容性，同时也增加了适配和配置成本，并可能限制工具对特殊场景的表达能力。

CLI 的分发方式也更符合现有软件工程习惯。一个工具可以被打包成 `.exe` 或其他可执行文件，也可以通过系统包管理器、语言包管理器或安装脚本发布。用户或 Agent 下载、安装后即可执行；版本管理、自动更新、日志、退出码、环境变量、权限控制和管道组合，都可以直接复用操作系统与既有工具链。CLI 不只服务于某一个 Agent 客户端，人、脚本和不同的 Agent 都可以调用同一个工具。

从这个角度看，CLI 的核心优势不是某条命令执行得天然更好，而是它站在一套已经被长期验证的通用工程基础之上。MCP 如果只是把同一项能力再包装成一层专供 AI 调用的接口，就可能产生重复建设。随着模型越来越擅长阅读 Skill、理解帮助文档、调用命令并处理执行结果，这层专用包装的必要性会继续下降。

不过，这个理由同样存在边界。CLI 并不天然保证可移植、一致、可扩展或自动更新：它仍然可能依赖特定操作系统、CPU 架构、运行时和外部依赖；很多 CLI 的文本输出主要面向人类，也并不稳定或适合机器解析。要让 CLI 真正适合 Agent，仍需要结构化输出、稳定的退出码、非交互模式和明确的版本兼容策略。

另一方面，MCP 的本地 `stdio` 服务本身也可以被打包成可执行程序，由客户端作为子进程启动；MCP Registry 也支持通过软件包分发本地服务。因此，“下载一个可执行文件就能运行”并不是 CLI 独有的能力。MCP 对输入和输出使用 JSON Schema 进行约束，在跨客户端互操作方面甚至可能比自由度过高的 CLI 更一致。协议约束既是成本，也是它存在的主要价值。

所以，更准确的论点不是“CLI 天生在所有方面都优于 MCP”，而是：CLI 能直接复用成熟、开放、同时面向人和机器的工程生态；如果 Agent 已经能够可靠理解和调用这些工具，那么为同一能力额外维护 MCP 接口的收益，可能不足以覆盖它带来的协议适配和配置成本。

#### 3. CLI 是人和 AI 共享的操作界面

CLI 还有一个非常重要的优势：它不只是给 AI 使用，也可以直接给人使用。同一条命令既能由 Agent 执行，也能由 Agent 告诉用户，再由用户复制到终端里手动执行。用户可以在执行前检查参数，在执行后查看输出，还可以重复运行同一条命令进行复现、核实和验证。

这种人机共享的界面会带来很强的可审计性。Agent 的操作不再是隐藏在专用协议后面的一次抽象工具调用，而是人类工程师本来就能理解和使用的命令。出现问题时，人可以脱离 Agent，直接在相同环境中重放命令、观察错误、缩小问题范围。文档、脚本、CI 和故障排查记录也都可以复用同一套调用方式。

MCP 的设计中心则是 AI 应用。MCP 工具通常由 Host 中的 Client 发现，并通过 `tools/call` 发给 Server。用户需要安装或进入支持 MCP 的 Host，才能通过界面、审批框或活动日志间接使用这些工具。离开对应客户端后，一个普通用户通常无法像执行 CLI 命令那样，直接在终端里调用、复现和核验同一次操作。

严格来说，“MCP 只能给 AI 使用”并不完全准确。普通软件或人为编写的客户端同样可以发送 MCP 的 JSON-RPC 请求，支持 MCP 的 Host 也可以向用户展示工具、要求确认并保留执行日志。区别并非人类在技术上绝对不能调用，而是 MCP 没有天然提供像 CLI 那样成熟、直接、普遍存在的人类操作入口。

因此，这条理由更准确的结论是：CLI 为人、Agent、脚本和 CI 提供了同一种可见、可复制、可重放的操作语言，而 MCP 主要提供面向 AI Host 的机器接口。前者在人机协作、故障排查和结果验证上具有结构性的优势。

#### 4. CLI 是大模型的能力压缩接口

我认为 CLI 的优势不会因为大模型继续进化而消失。无论模型的推理能力、上下文窗口和自主性发展到什么程度，它始终需要一种低成本调用外部能力的方式。CLI 的优势非常明确：大模型只需要生成一条很短的命令，就能让一个成熟工具完成大量实际工作。

一条命令背后可以封装复杂的搜索、计算、网络请求、文件处理、构建、测试和部署流程。模型不必在上下文中重新描述每一个步骤，也不必用 token 模拟本应由程序完成的确定性计算。它只需要表达目标和关键参数，再读取必要的执行结果。CLI 因而像一种能力压缩接口，让模型用更少的 token 驱动更大的工作量，完成更多事情。

这种优势不会简单地被更大的上下文窗口抵消。即使未来 token 更便宜、模型能读取更多内容，减少无关信息仍然意味着更低的延迟、更少的注意力干扰和更清晰的执行边界。更重要的是，模型能力再强，也没有必要重复实现已经存在且经过验证的工具。把确定性工作交给程序，把判断、规划和异常处理留给模型，仍然会是更经济的分工。

当然，CLI 只有设计良好时才能真正节省 token。如果命令缺少稳定的非交互模式，帮助文档混乱，或者输出大量面向人的冗余文本，模型仍会浪费上下文去猜测参数和解析结果。因此，面向 Agent 的 CLI 应当支持简洁的帮助信息、结构化输出、安静模式、明确的退出码和可预测的错误格式，再通过 Skill 按需提供使用说明。

所以，CLI 的长期优势不只是“终端工具历史悠久”，而是它提供了一种稳定的认知分工：模型负责理解意图和作出决策，CLI 负责以确定性的方式执行被压缩后的复杂能力。只要计算资源、上下文和注意力仍然有成本，这种分工就会持续存在。

### 驾驭工程（Harness Engineering）

当模型能力逐渐成熟，并且已经能够通过 CLI 和 Skills 使用外部工具之后，问题就从“模型能不能做”转向了“怎样让模型长期、稳定、可控地把事情做完”。这正是驾驭工程需要解决的问题。

驾驭工程关注的不是单次提示词，也不只是模型本身，而是围绕模型建立的整套运行环境与约束机制，包括上下文管理、工具组织、权限控制、任务循环、状态保存、错误恢复和结果验证。同一个模型被放进不同的 Harness 中，最终表现可能出现巨大差异。

#### 为什么叫“驾驭工程”

一开始，大家对“驾驭工程”这个概念都很陌生。我第一次看到 Harness Engineering 这个词时，也觉得有些奇怪。但从字面上理解，它讨论的其实就是怎样真正用好一个强大的工具。

随着使用越来越深入，我反而觉得“驾驭”这个词非常准确。AI 本身拥有能力，并不意味着它能够自动、稳定地把事情做好。怎样让 AI 在正确的环境里运行，怎样让它理解任务、调用工具、持续执行，并最终交付可以验证的结果，都需要人为设计。

这不是写好一句提示词就能解决的问题，而是一种体系化思考。我们需要同时考虑模型得到什么信息、可以使用哪些工具、每一步如何推进、出错后怎样恢复、结果由谁验证，以及任务状态怎样持续保存。驾驭工程，就是把这些原本零散的实践组织成一套完整的工程方法。

Harness 的核心就是“驾驭”。AI 是一个能力很强的工具，但如果想让它更有效地执行任务，仅仅把目标告诉它还不够。我们需要给它明确的方向、可以行动的边界和持续反馈，让它在整个执行过程中始终沿着正确的道路前进，并尽可能准确地完成每一步。

所以，驾驭工程追求的不是让 AI 偶尔产生一次正确结果，而是让正确执行变得稳定、可重复。任务一旦变长，模型就可能遗忘目标、误解约束、跳过步骤或在局部问题上越走越远。Harness 要做的，就是及时发现这些偏离，通过状态记录、阶段检查和结果验证把 AI 拉回正确路径。

#### 熟练使用最终会自然形成架构

更直接地说，一个真正熟练使用 AI 的人，最终会自然而然地形成一套有形、有效的架构。他会把自己反复验证过的提示、规则、工具和工作流固化下来，而不是每次面对新任务时都从零开始教 AI。这些结构可能最初只是几个文件或脚本，但随着实践积累，会逐渐发展成可管理、可复用的个人 Harness。

我在《[为什么你需要一个自己的 Agent Skills 仓库](./why-personal-agent-skills-repo.md)》中已经实践了这个观点。我把不同类型的 Skills 组织成多个插件，放在 GitHub 项目中通过 Git 管理版本。每个插件可以独立演进，我也可以根据当前任务随时安装、启用或停用需要的能力。核心 `SKILL.md` 只维护一份，再针对不同 Agent 平台增加适配层。

这种架构让 Skills 不再依附于某一次对话、某一台设备或某一个 AI 平台。换电脑、切换 Agent、进入不同项目时，我仍然可以从仓库恢复自己的技能体系。Git 提供版本、变更历史和回滚能力，GitHub 提供分发与同步入口，而插件划分则控制每次实际加载的能力范围。

这就是驾驭工程的一个具体形态：把“我知道怎样让 AI 做好这件事”从个人记忆变成可以被版本控制、按需装配和跨平台分发的工程资产。模型可以更换，设备可以更换，工作环境也可以变化，但经过沉淀的使用方法不会随之丢失，也不容易在长期使用中发生无意识的偏离。

不过，Git 主要保证的是 Skill 定义、脚本和配置的可迁移性，并不能自动消除所有环境差异。运行时版本、系统依赖、密钥、权限和网络条件仍然可能造成行为偏差。因此，完整的驾驭工程还需要把依赖声明、环境检查、私密配置边界和验证流程一起纳入管理。可移植不是一句口号，而是需要持续维护的工程属性。

从工具调用和能力暴露的视角看，我认为 MCP 服务的本质其实就是一种给 AI 使用的 OpenAPI。它把外部系统具有哪些能力、需要哪些参数、会返回什么结果，以一种模型能够发现和调用的方式描述出来。过去的 OpenAPI 主要服务于开发者和应用之间的集成，而 MCP 只是把接口的主要使用者换成了 AI Agent。

换句话说，MCP 并没有创造一种全新的能力，它更像是将已有的接口描述、服务发现和远程调用模式重新包装，使其适应 Agent 时代。它的重要性来自使用者发生了变化，而不一定来自底层工程范式发生了根本变化。

### Loop：早已存在的任务循环

最后再来看 Loop。循环这个理念本身非常好，但它并不是 AI 时代突然出现的新发明。早在我所说的“古法编程”阶段，软件工程中就已经存在类似的循环。GitHub Issues 以及更早的项目管理系统，本质上都在围绕任务建立持续运转的状态机制。

一个用户提出问题、Feature 或具体需求后，团队会创建一条 Issue。Issue 记录任务的目标、背景和约束，并通过待处理、进行中、等待审核、已完成等状态管理任务的生命周期。它既是一次工作的触发源，也是执行者理解“为什么要做这件事”的意图载体。

过去，这个循环主要由人驱动。人查看 Issue，理解需求，修改代码，运行测试，提交结果，再根据评审或反馈继续修正。任务状态推动人采取下一步行动，新的执行结果又反过来更新任务状态。所谓 Loop，早就存在于这套“读取任务、执行工作、检查结果、更新状态”的往复过程里。

AI 时代真正发生的变化，不是循环被发明了，而是循环中的执行者开始从人变成 Agent。Issue 仍然可以作为任务入口和状态载体，只是现在可以由 AI 读取任务、调用工具、修改代码、运行验证并回写结果。过去依靠人的责任感、经验和注意力维持的循环，开始被显式地写进 Agent 的运行机制中。

#### 从 Issue 到 PR 的 Agent Loop

从 AI 时代来看，Issue 不仅记录任务，也回答了 Agent 为什么要查看任务、什么时候应该开始执行。新的 Issue、标签变化或状态变化都可以成为动态触发信号，让 Agent 主动进入工作循环，而不是每次都等待人重新输入一段提示词。

Agent 读取 Issue 后理解需求，在独立分支中修改内容并执行验证，再把执行结果、变更内容和测试证据关联回原来的 Issue。随后它可以创建 PR，交给自动化检查或另一个 Agent 审查。审查发现问题后，Agent 继续修复和验证；满足合并条件后再合并分支，并同步更新或关闭 Issue。

```text
Issue 触发
    ↓
Agent 理解并执行任务
    ↓
提交代码与验证结果
    ↓
创建 PR 并接受审查
    ↓
修复问题、重新验证
    ↓
合并分支、更新 Issue
    ↺
```

这是一种正向循环：每次执行都会产生代码、测试、评审意见和状态记录，这些结果又成为下一轮行动的输入。项目可以在循环中持续迭代和优化。

#### 把人的研发循环复刻给 Agent

从人的视角看，软件开发本来就是一个循环。开发者不断理解需求、编写代码、测试、评审、修复和发布，再根据新的问题开始下一轮迭代。人的工作时间、注意力和精力有限，这个循环必然会暂停，也需要频繁重新进入上下文。

Agent 则可以全天候执行。我们应该把人类已经验证过的研发循环复刻为 Agent 可以读取和推进的状态机制，让它知道何时启动、当前处于哪一步、完成的标准是什么，以及接下来应该做什么。这就是为什么 AI 时代需要引入 Loop 工程：不是简单地让模型重复执行，而是让 Agent 能够被事件触发，沿着明确的状态流主动推进任务。

不过，全天候运行不等于无限循环。一个可靠的 Agent Loop 必须同时定义触发条件和停止条件，并设置重试上限、幂等规则、资源预算与权限边界。连续失败、需求冲突或超出授权范围时，流水线应当自动停止并上报，而不是继续重复消耗资源、持续放大错误。

因此，Loop 工程的目标不是让 Agent 永不停下来，而是让它在无需人类持续盯守的情况下，能够主动开始、持续推进、验证结果，并在完成任务或遇到边界时正确停止。

#### 真正的 Loop 不应依赖人工审批

我不认为 Agent 的正常 Loop 应该依赖人工审批。如果每执行一轮都必须等待人点击确认，整个循环就会被人的工作时间和注意力重新限制。这样的系统仍然只是 AI 辅助下的人工流程，而不是真正自治的 Agent Loop。

从 AI Agent 时代的视角看，Loop 应当采用流水线模式。任务进入流水线后，需求解析、代码修改、测试、审查、修复、验证和合并判断都应该由 Agent 与自动化规则持续推进。一个阶段的输出直接成为下一阶段的输入，成功则向前流转，失败则按规则重试、回滚或终止，不需要人在每个步骤中同步介入。

人的角色应该从“逐次审批执行”转向“预先设计系统”。人负责定义权限范围、质量标准、资源预算、验收规则和停止条件；Agent 在这些边界内自主运行。这样既不会让人工审批阻塞循环，也不会把“自治”误解为毫无约束地无限行动。

这里仍需保留一个边界：涉及资金、生产数据、对外发布或其他不可逆操作时，系统不能凭空获得超出用户授权的权力。但这不意味着必须在每一轮临时审批。更合理的方式是预授权、沙箱隔离、策略门禁和分级权限；一旦越界，流水线自动停止并交还控制权。人工处理的是异常和授权边界，而不是正常流程中的每一步。

因此，理想的 Agent Loop 应当是无需人工在环的自治流水线：人设计轨道和规则，Agent 负责在轨道内持续运行。

### 这些概念都不是突然出现的

从人类已有的工程经验来看，Agent 时代的这些概念其实都有迹可循。它们并不是某一天突然爆发，也不是某个人凭空发明了一个全新的体系。Skills、CLI、驾驭工程和 Loop，都能在过去的软件工程、项目管理、自动化工具和人机协作中找到对应的原型。

AI 时代真正带来的变化，是模型开始成为可以读取信息、调用工具、维护状态并持续执行任务的新主体。原本由人完成的经验传递、工具调用、流程推进和循环迭代，需要被重新组织成适合 Agent 使用的工程结构。因此，这些概念的出现不是偶然，而是 AI 能力发展到相应阶段后的必然趋势。

很多时候，并不是某个概念刚刚诞生，而是终于有人找到了一个准确的词，把早已分散存在的实践描述出来。一个词一旦足够准确，大家就能围绕它交流、总结和继续建设。Harness Engineering 和 Agent Loop 的流行，更多是在为已经发生的变化命名，而不是从零创造变化本身。

在过去的时代，我们甚至不会专门给这些东西起名字。它们只是软件工程中默认存在的做事方式，或者是一个熟练工程师自然而然会完成的工作。人会主动补充缺失信息、记住任务状态、选择合适工具、发现错误并调整方向，所以这些能力长期隐藏在人的经验和责任感里，不需要被单独抽出来描述。

当执行主体逐渐从人变成 AI，这些原本隐性的环节就不能继续依赖人的本能。它们必须被写进文件、协议、状态机、工具链和验证规则，成为机器可以读取和执行的显式结构。正是在这个显式化的过程中，我们才开始需要 Skills、Harness 和 Loop 这些词。

因此，新名词的出现往往不代表新机制凭空诞生，而代表一种过去无需言明的工程常识，现在必须被准确描述、系统设计并交给 Agent 执行。

理解这一点后，我们就不必追逐每一个突然流行的新名词，而应该观察它在解决什么旧问题、继承了哪些工程传统，又因为 AI 这个新执行主体发生了什么变化。只有这样，才能看清哪些形态只是阶段性的中间产物，哪些能力会长期保留下来。
