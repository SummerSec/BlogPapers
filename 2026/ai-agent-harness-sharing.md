## 第二章：可进化的 Harness

这是整场分享的主体。

### 为什么需要 Harness

单个 Skill 写出来以后，还会遇到：

- 多个平台怎么使用
- Skill 多了怎么组织
- 修改以后怎么同步
- 不同版本怎么管理
- 效果变差怎么回滚
- 如何根据真实结果继续改进

因此，Harness 解决的是 Skills 的完整生命周期。

### Harness 是什么

> Harness 是一套围绕 Skills 建立的管理、分发、运行、验证和进化系统。

```text
Harness
├── Skill 组织与管理
├── 多平台分发与适配
├── Git 版本与发布
├── CLI / Tools 执行能力
└── Eval / Trace / Loop 进化机制
```

### 可分发

核心 `SKILL.md` 只维护一份，
在外层适配 Claude Code、Codex、Cursor 等平台。

重点不是复制文件，而是：

> 一份核心能力，多个稳定入口。

### 可管理

按领域组织成 Plugin，需要什么安装什么、启用什么：

```text
开发工具 / 安全研究 / 内容写作 / 部署运维
```

避免所有 Skill 同时进入上下文，也避免不同领域互相干扰。

### 可版本化

使用 Git 管理：

- 修改历史
- Skill Diff
- 版本号
- 上游来源
- 发布记录
- 回滚位置

中心句：

> 没有版本记录的 Skill，只是一份会不断变形的提示词。

### 可进化

Skill 不可能一次写对，需要通过真实任务持续修正：

```text
执行任务
  ↓
收集结果与轨迹
  ↓
定位重复失败
  ↓
生成候选修改
  ↓
重新评测
  ↓
接受或回滚
```

这里直接包含原来的评测、轨迹和 Loop，不再单独拆章节。

关键原则：

- 结果告诉我们“错没错”
- 轨迹告诉我们“为什么错”
- 确定性规则负责诊断
- LLM 只生成候选修改
- 验证通过后才接受
- 失败修改进入黑名单
- 指标下降自动回滚
- 长期没有提升就停止

### Skill 的语义边界

“漏洞”换成“风险”，准确率从 89.3% 降到 62.1%，
说明 Skill 的质量不仅取决于流程，也取决于关键词的语义宽度。

这部分作为 Harness 的质量控制案例：

> Skill 能被管理和迭代，还不够；
> 它必须经过评测，才能知道修改是否真的有效。

## 第三章：AI 时代的个人与团队竞争力

### 先改变做事逻辑，不是只更换工具

不要在旧流程旁边加一个聊天框，而要重新思考 AI 能做什么、
需要什么上下文、如何执行和验证。

### AI 应该成为基础设施，而不是一盏灯

真正的改变不是增加 AI 插件，而是围绕 AI 重做流程、
组织方式和验证入口。

### 给 AI 留下观察和验证窗口

日志、API 记录、测试结果和运行环境应该能被 AI 直接读取。
AI 不只生成代码，也要参与定位和验证。

### 先做可用 Demo，快速验证真实需求

AI 降低了开发成本，但没有降低维护成本。
先验证“有没有人用”，再决定是否继续建设。

### 优先复用成熟能力

先找现有项目、标准库、平台能力和稳定工具。
AI 让造轮子更容易，也让无意义的维护负担增长得更快。

### 超级个体与专家 Review 应该共存

AI 让个人可以跨角色做原型，但不等于拥有所有领域的专业判断。
个人负责推进，专家负责关键风险检查。

### 真正的竞争力是自我修正速度

功能、代码和想法都会被复制。
别人难以复制的是更快学习、验证、推翻旧判断并继续前进的能力。
