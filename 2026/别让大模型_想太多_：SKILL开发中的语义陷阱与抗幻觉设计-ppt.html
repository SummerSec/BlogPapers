Semantic Trap / Skill Design / Anti-Hallucination

别让大模型
“想太多”

SKILL 开发中的语义陷阱与抗幻觉设计。不是流程不够严，而是某些关键词在模型语义空间里，天生就更容易把边界撑开。

漏洞版正确率 89.3%

语义边界更窄，约束更容易落地。

风险版正确率 62.1%

仅替换一个词，就出现大规模越界输出。

核心结论 27pt

Prompt 的边界，不只由规则决定，也由词义宽度决定。

Problem Statement

一个词，为什么能拉开 27 个百分点？

共同前提

两套 Skill 的流程、文件结构、工具链完全一致

唯一差异只是核心术语：一个写“漏洞”，一个写“风险”。

实验样本
56

同一批营销接口、同一模型环境、同一审计目标。

引发的问题

明明约束写清了，模型为何还会“自由发挥”？

这不是个别失误，而是一个可重复出现的系统性现象。

抗幻觉不只是“把规则写得更细”，还包括“把关键词选得更窄”。
Experiment Result

同一任务，不同术语，输出质量骤变

Claude Code DeepSeek V3.2 营销接口测试集
biz-vul-security
89.3%

关键词是“漏洞”，模型聚焦于定义内的漏洞类别。

biz-risk-security
62.1%

关键词换成“风险”，模型开始输出更多定义外的内容。

差值：27 个百分点
Task Context

案例本身并不复杂，复杂的是模型如何理解边界

业务场景

  • 目标接口是一个“端外券领取”接口：outflow/voucher/receive。
  • 用户点击“领券”后，系统会调用该接口发放营销奖励。
  • 审计核心是：攻击者能否改参数，越权领取不属于自己的奖品。

正确答案

  • campId 这类关键营销参数来自服务端配置，不由请求传入。
  • 用户可控参数与营销活动绑定关系不直接相关。
  • 因此结论应为：无安全隐患 / 无漏洞。

也就是说，问题并不在“代码太难懂”，而在“模型会不会忠实待在定义好的审计边界里”。

Report A

“漏洞”版：逻辑链条收敛，判定边界清晰

biz-vul-security

核心词：漏洞
是否为营销接口：是 任意发奖漏洞判定结果：无漏洞 1. campId 从配置获取 2. 非外部可控 3. 结论：无漏洞

为什么它表现稳定

  • 它严格围绕“漏洞定义文件”中的判定标准执行。
  • 输出是一个窄边界的“是/否 + 依据”过程。
  • 即使看到别的代码瑕疵，也不会主动扩展评估范围。
Report B

“风险”版：不是算错，而是开始越界思考

biz-risk-security

核心词：风险
是否为营销接口：否 接口风险等级：低风险 1. 代码逻辑错误 2. 参数校验不足 3. 权限控制缺失

偏移点

  • 先把营销接口识别错了。
  • 再发明出定义文件里根本不存在的问题类别。
  • 这些内容并非完全胡编，但它们不属于当前 Skill 定义的审计目标。
False Lead

不是“风险版写得差”，而是机械替换后仍然失控

我们对两个 Skill 的全部文件做了 diff，确认没有结构性差异。

主文件、定义文件、示例文件、步骤框架全部只是做了术语替换：vulnerability -> risk。

这一步排除了什么

  • 不是漏写规则。
  • 不是工具链不同。
  • 不是 Few-shot 示例数量不同。
  • 不是执行步骤不一致。

当所有“工程差异”都被排除后，唯一剩下的解释，只能落在词汇本身的语义边界上。

Methodology

高质量 Skill，本来已经有一套很强的约束体系

01
结构化工作流

把任务拆成不可跳过、不可乱序的步骤跑道。

02
校验-执行-验证

每一步都写状态、回读状态，降低遗忘与偷工减料。

03
定义文件 + 示例

用穷举边界和反误判案例，把解释空间压到最小。

04
领域定制工具

减少模型面对通用工具时的选择噪声。

Workflow

第一根支柱：把任务变成确定性的执行跑道

1

Progress 文件创建

先建立外部状态，后续每一步都能回看。

2

链路追踪分析

识别接口入口与核心防御机制。

3

参数流向分析

追踪关键参数是否可被外部控制。

4

接口分析

识别是否属于营销接口场景。

5

漏洞评估判定

只能按定义文件中的漏洞类型做判断。

6

Final Answer 报告

输出结构化结论，而不是开放式作文。

Validation Loop

第二根支柱：校验 → 执行 → 验证，强迫模型“回头看”

  • 前置校验：读取进度文件，确认前一步已完成。
  • 执行过程：记录将执行的动作与中间结果。
  • 后置校验：再次回读状态，验证结果确实被写入。

这相当于给大模型外挂了一层“外部记忆”，不再完全依赖上下文窗口里的隐式记忆。

Definition File

第三根支柱：穷举边界，纠正常见误判

定义文件做什么

  • 明确“什么算漏洞，什么不算漏洞”。
  • 罗列无漏洞场景，减少模型自行脑补。
  • 把容易误判的场景提前写成反例。
参数从配置获取 campId / prizeId 来自可信配置 不从请求参数传入 => 非外部可控 => 无漏洞
Custom Tooling

第四根支柱：领域定制工具，不给模型太多犹豫空间

通用工具的问题

  • 能搜、能读、能执行，但不告诉模型什么时候用、为什么用、怎么用。
  • 模型很容易在工具选择上反复摇摆，甚至挑错入口。

领域工具的价值

  • 把“使用时机”和“参数形态”提前编码进 Skill。
  • 模型只需沿着预设路径执行，探索空间显著缩小。
Transition

问题来了：四大支柱都在，为什么换成“风险”就失灵？

同样的流程、同样的校验、同样的定义、同样的工具，换了一个词之后，模型仍然能把边界撑开。

这说明失控不只发生在“执行层”，还发生在更底层的“语义激活层”。

Semantic Trap

什么是“语义陷阱”

在人类看来意思相近的词，在大模型的语义空间里，可能激活出完全不同宽度的联想网络。

它和普通“提示词不够清晰”不同

这里不是让指令从模糊变清楚，而是在已经写得很清楚的前提下，某个关键词仍然悄悄扩大了认知范围。

核心症状

  • 模型开始考虑定义外的内容。
  • 约束条款看似还在，实际约束力下降。
  • 输出不是随机错误，而是“有理由地越界”。
Word Boundary

“漏洞”与“风险”，在模型里并不是等价近义词

漏洞 / Vulnerability

窄边界
  • 具体、可利用、可验证。
  • 天然适合“有 / 无”的二元判断。
  • 注意力容易收束到定义文件中的少数具体类型。

风险 / Risk

宽边界
  • 偏可能性、趋势与程度。
  • 天然带有“未来可能发生什么”的展望性。
  • 容易把代码质量、权限设计、参数充分性一起卷进来。
Error Pattern

风险版的错误，不是一个点，而是三种稳定模式

68%
范围溢出

输出定义文件中根本不存在的风险类型。

24%
等级虚高

把本应“无风险”的情况说成“低风险”。

8%
逻辑偏移

分析路径本身被宽词带偏，先把业务场景理解错。

Pattern Breakdown

关键特征：它不是在框架内算错，而是在框架外思考

范围溢出
  • 参数校验不充分风险
  • 权限控制缺失风险
  • 代码逻辑错误风险
  • 配置变更风险
等级虚高

只要看到“不够完美”的实现细节，就倾向于归类为低风险，即便它并不命中任何定义条目。

逻辑偏移

同一段代码里，“漏洞”版能识别为营销接口，而“风险”版先把场景理解错，后续判断自然全部偏移。

Analogy

“破损”与“有问题”的差别，就是“漏洞”与“风险”的差别

给实习生两份指令

  • 版本 A：检查仓库里是否有“破损”的货物。
  • 版本 B：检查仓库里是否有“有问题”的货物。

如果清单完全一样，A 会集中找物理破损，B 却可能顺手报告标签不齐、货架积灰、摆放不规范。

Common Trap Pairs

这不是个案，很多近义词都会触发类似问题

检查 → 审查

从客观判定，滑向主观评价。

列出 → 描述

从结构化输出，滑向解释性发挥。

缺陷 → 问题

从具体缺陷，滑向泛化改进意见。

总结 → 分析

从提炼已有信息，滑向推理与假设。

要求 → 建议

从明确边界，滑向开放联想空间。

错误 → 异常

从明确错误，滑向模糊偏差与轻微波动。

Why This Matters

传统 Prompt 工程是在砌围墙，语义边界控制是在检查砖会不会膨胀

  • 格式约束、角色设定、Few-shot，本质上都在加强“围墙”。
  • 但如果关键词自身语义宽度过大，它就像会膨胀的砖。
  • 围墙再严，只要砖自己向外长，边界还是会被撑裂。
Practical Principle

实战原则：在表意准确的前提下，始终优先窄边界词

优先名词目标

“检测安全漏洞”通常优于“评估安全性”。

优先动宾结构

“列出未处理异常”通常优于“分析异常处理情况”。

先测同义词宽度

把候选词放进统一句式里，看看它会诱发多少种合理输出。

Semantic Drift Tests

写完 Skill 后，至少做三类漂移测试

1

排除测试

问模型：除了定义内条目，还有哪些也算 X？如果它能轻易说出很多额外内容，说明词太宽。

2

最小对比测试

挑一个你确定无问题的样例，看输出是否仍夹带“定义外”附加判断。

3

换词对照测试

把核心词换成更窄的同义词，在同样测试集上重新跑，看结果怎么变。

Boundary Anchoring

如果业务上必须使用宽词，就给它补三道锚

前置否定清单

直接写出“哪些虽然通常也算风险，但本 Skill 一律不评估”。

输出格式硬约束

把输出做成枚举字段和填空式结构，不给“额外发现”留口子。

反例强化

把“看起来像，但其实不算”的场景写进 Few-shot 里，给模型画红线。

Checklist

一个 Skill 完成后，至少问自己这 8 个问题

  • 核心术语是不是最窄边界？
  • 排除测试能不能稳住？
  • 有没有明确的“不包括”清单？
  • 输出是结构化填空，还是开放作文？
  • 有没有“表面像但实际上不算”的反例？
  • 最小对比测试会不会冒出额外内容？
  • 有没有外部状态来对抗遗忘？
  • 工具有没有足够领域化，帮助模型收敛？
Tooling Up

从理论走向工具：语义陷阱检测器

为什么需要它

当 Skill 变长、引用文件变多、术语埋得更深时，靠人工逐词巡检既慢又容易漏。

它做什么

  • 把高危词对固化成词典。
  • 把排查过程变成确定性步骤。
  • 不仅指出问题，还给替换和锚定建议。
Detector Workflow

检测器把“语感判断”变成可重复执行的四步流程

1

加载词典

读取已知语义陷阱词汇对与边界锚定知识。

2

提取关键词

核心术语、动作指令、约束词、输出描述词。

3

匹配与上下文分析

既看词本身，也看它在句子里扮演什么角色。

4

输出报告

给出风险等级、替换建议和边界锚定方案。

Example

最小示例：两次替换，就能把句子从危险区拉回安全区

原句

请审查这段代码中的安全问题， 只关注定义文件中的问题类型。

替换后

请检查这段代码中的安全漏洞， 只关注定义文件中的漏洞类型。

为什么“审查”危险

它更容易触发主观评价和额外意见，而不是通过 / 不通过式检查。

为什么“问题”危险

它的范围极宽，改进建议、代码风格、设计缺陷都可能被包进去。

Process Integration

最佳落点：把语义检测做成 Skill 开发流程里的标准环节

Skill 编写 语义陷阱检测 修复 / 锚定 功能测试 发布

它相当于给 Prompt / Skill 增加了一层“语义级别的静态分析”，把原本依赖经验的语感判断流程化。

Takeaway

先审词，再调流程

89.3% 对 62.1%，差别不在模型突然变笨，也不在流程突然失效，而在于“风险”这个词比你以为的更宽。
流程仍然重要

步骤、定义、示例、工具，都是必要的约束层。

但词更底层

关键词决定模型会激活多大的认知空间。

行动建议

下次写 Skill 时，先问一句：这个词在模型眼里，真的和我想的一样窄吗？
