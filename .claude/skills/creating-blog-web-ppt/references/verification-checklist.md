# 验证清单

没有新的验证证据之前，不要声称完成。

## 执行顺序

必须按这个顺序验证：

1. 先做静态检查
2. 再做浏览器检查
3. 最后做收尾清理

## 静态检查

1. 读取最终 HTML，确认：
   - 存在 `<title>`
   - 存在多个 `section.slide`
   - 图片路径相对文章目录正确
   - favicon 路径与 deck 所在位置匹配
   - 若存在 `SUMSEC` / `sumsec` 品牌位，则其为可点击链接而不是纯文本
   - 品牌位 `href` 指向原文公开地址，而不是仓库地址或当前 PPT 地址
   - 右下角存在可见的全屏按钮，且具备 `aria-label`
   - 主画布存在明确的视口约束策略，如 `max-width`、`max-height`、`aspect-ratio`、断点降级或等效实现
   - 原文图片存在受限的 figure 容器或等效约束，而不是裸图直接参与撑开布局
   - 原文内容图默认优先使用 `contain` 或等效的完整显示策略，而不是默认裁切
2. 对最终 HTML 运行 `ReadLints`。

## 浏览器检查

从仓库根目录启动一个临时静态服务，再用浏览器打开最终页面。

至少确认：

- 页面能正常打开
- 页面标题正确
- slide 数量大于 0
- 初始页码正确
- 执行一次翻页操作后页码发生变化
- 点击右下角全屏按钮后，页面能进入全屏或在不支持环境下表现为明确降级
- 首屏 slide 在当前浏览器视口内完整可见，没有关键内容被裁掉
- 页面没有横向滚动条
- 原文大图没有把 figure 容器、列布局或主画布撑破
- 缩小窗口后主画布仍未超出视口，或已退化为纵向兜底布局
- 若页面存在 `SUMSEC` 品牌位，点击后能打开原文页面
- 浏览器控制台没有报错

## 最低证据格式

验证证据至少要覆盖：

- 打开的 URL
- 页面标题
- 当前视口尺寸
- slide 数量
- 翻页前页码
- 翻页后页码
- 全屏按钮结果
- 是否横向溢出
- 图片容器约束结果
- 首屏是否完整可见
- 品牌位跳转结果
- 控制台错误数量

建议按下面这个格式回报：

```markdown
验证结果：
- 打开的 URL：`...`
- 页面标题：`...`
- 当前视口：`...`
- slide 数量：`...`
- 翻页前页码：`...`
- 翻页后页码：`...`
- 全屏按钮结果：`成功 / 已优雅降级`
- 横向溢出：`否`
- 图片容器约束：`已限制在固定 figure 内`
- 首屏完整可见：`是`
- SUMSEC 跳转：`成功 / 不适用`
- 控制台错误：`0`
```

## 验证失败时怎么回报

如果验证没有通过，不要说“已完成”。

至少要明确回报：

- 失败发生在哪一步
- 实际看到的现象
- 是否已经定位到原因
- 下一步准备怎么修

建议按下面这个格式回报：

```markdown
当前还不能算完成。

验证失败点：
- 步骤：`浏览器检查`
- 现象：`翻页后页码没有变化`
- 原因判断：`疑似页码更新逻辑未绑定当前索引`
- 下一步：`修复脚本后重新打开页面验证`
```

## 收尾清理

验证完成后：

- 关闭临时静态服务
- 删除本次验证产生的临时浏览器产物

## 这些情况不能算完成

- 浏览器没有打开最终的同目录 HTML
- deck 依赖一个已经删除的中间文件
- 控制台还有报错
- 本地静态服务下图片路径失效
- 翻页后页码没有变化
- 存在全屏按钮但点击无效，且没有降级说明或禁用态
- `SUMSEC` 品牌位存在但不能跳到原文
- 首屏主画布超出视口导致内容被裁切
- 页面出现由 deck 主画布导致的横向滚动
- 原文大图把 figure 容器、列布局或固定舞台撑破
- 原文图表被默认裁切，导致关键内容不可见
- 小屏下没有响应式降级，导致内容溢出或不可读
