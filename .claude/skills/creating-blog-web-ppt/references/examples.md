# 示例

## 示例 1：把仓库文章做成网页版 PPT

用户请求：

```text
把 @2026/某篇文章.md 做成网页版 PPT
```

期望执行：

1. 读取源 Markdown
2. 读取仓库视觉参考：
   - `assets/css/style.scss`
   - `_layouts/default.html`
3. 写出 `visual thesis`、`content plan`、`interaction thesis`
4. 生成同目录、同 basename 的 HTML
5. 保持文章图片相对路径可用
6. 做浏览器级验证

期望输出路径：

```text
2026/某篇文章.html
```

## 示例 2：文章很长，不强压页数

用户请求：

```text
把这篇长文做成网页版 PPT，页数不限
```

正确做法：

- 不强行压成 10 到 12 页
- 按观点分组拆页
- 保持单页信息密度可扫读
- 可以做成长版 deck

错误做法：

- 为了“像 PPT”硬压成太少页面
- 导致每页塞满长段文字

## 示例 3：图片已经在文章目录里

Markdown 中已有：

```text
![示意图](./pic/llm/02-accuracy-dashboard.png)
```

如果 HTML 与文章同目录：

- HTML 中也应继续使用 `./pic/llm/02-accuracy-dashboard.png`
- 不要无故改成别的根路径

## 示例 4：用户只说“做个 HTML 演示页”

如果用户没有指定路径：

默认解释为：

- 单文件 HTML
- 同目录输出
- 同 basename
- 不更新首页和归档

## 示例 5：用户覆盖默认规则

用户请求：

```text
把这篇文章做成网页版 PPT，但文件名不要和原文同名，放到 resources/ 下面
```

正确做法：

- 明确识别这是用户在覆盖默认规则
- 不再坚持“同目录 + 同 basename”
- 只把本 skill 的其他未冲突约定继续沿用

这类情况下仍可沿用：

- 单文件 HTML
- 仓库视觉风格
- 浏览器级验证

## 示例 6：这些情况不要触发本 skill

用户请求：

```text
帮我调整这篇博客的段落间距和图片大小
```

正确做法：

- 不触发本 skill
- 这是普通博客排版修改，不是“文章转网页版 PPT”

用户请求：

```text
给首页 README 增加这篇文章的入口
```

正确做法：

- 不触发本 skill
- 这是首页维护，不是独立 deck 生成

## 示例 7：完成前必须验证

最低验证动作：

1. 本地静态服务打开最终 HTML
2. 读取页面标题
3. 读取 slide 数量
4. 记录翻页前页码
5. 执行一次翻页
6. 记录翻页后页码
7. 检查控制台错误

## 示例 8：最终汇报模板

```markdown
未处理项 / 剩余风险：
- ...
```
